# One File to Orient Any Assistant

`AGENTS.md` is a single, self-contained markdown file you drop at the root of your data folder вҖ” `~/.wolffish/AGENTS.md` вҖ” to **orient any AI assistant you point at your Wolffish installation**. Open `.wolffish` in an editor, or hand the folder to a coding agent (Claude Code, Cursor, an MCP-connected assistant), and that assistant can read this one file to instantly understand what Wolffish is, where everything lives, and how to help you.

<Note>
  Don't confuse this with [`agents.md`](/configuration/agents-md) (lowercase, at `brain/prefrontal/agents.md`). That file tells **your Wolffish agent** how to do its job. **`AGENTS.md`** (uppercase, at the folder root) tells a **visiting** assistant how to navigate *your workspace*. Different audience, different file.
</Note>

## Why drop it in

A Wolffish workspace is just files вҖ” readable, editable, yours. That's the whole point: you can bring any AI to help you read, repair, or extend it. But a cold assistant doesn't know that `conversations/` lives under `brain/`, that capabilities are dot-prefixed, or that `cortex.db` is disposable. `AGENTS.md` front-loads all of that so the assistant is useful on the first message instead of the tenth.

Moments it pays off:

- *"Why did my agent do that?"* вҖ” the assistant knows to read `brain/corpus/` and `brain/prefrontal/.debug/`.
- *"Change my agent's personality."* вҖ” it edits `brain/identity/soul.md`, not random code.
- *"What am I spending?"* вҖ” it sums `usage/daily/*.md`.
- *"Add a tool that checks the weather."* вҖ” it scaffolds a capability the right way (see below).

## Creating a capability вҖ” the short version

One of the most common things you'll ask a visiting assistant is *"teach Wolffish a new skill."* The full walkthrough вҖ” with a complete, **working** Weather example that needs no API key вҖ” is in **[Creating Capabilities](/capabilities/creating-capabilities)**. But here's the entire idea in four steps, which is roughly what `AGENTS.md` tells the assistant:

<Steps>
  <Step title="Make a folder">
    `~/.wolffish/workspace/brain/cerebellum/<name>/`
  </Step>
  <Step title="Write SKILL.md">
    YAML frontmatter (`name`, `description`, `triggers`, `tools`) plus a markdown body of instructions. Add `danger_patterns` / `confirm_patterns` **only** if the tool can cause harm.
  </Step>
  <Step title="Add plugin/index.mjs (optional)">
    For custom logic, export `{ name, tools, execute(toolName, args) }` returning `{ success, output }`. Pure skills that lean on existing tools (`shell_exec`, `file_read`) skip this entirely.
  </Step>
  <Step title="Restart Wolffish">
    The cerebellum discovers the folder, the `triggers` make it matchable, and the `tools` become callable. No app rebuild.
  </Step>
</Steps>

That's the whole extension model: **a folder of markdown teaches your agent a new trick.** See [Writing Plugins](/capabilities/writing-plugins) and [Safety Patterns](/extending/safety-patterns) for the depth.

## Where it lives

Wolffish bundles this guide and writes it to the **root** of your data folder, refreshing it on every launch:

```bash
~/.wolffish/AGENTS.md
```

It sits next to `workspace/`, `runtime/`, and `logs/` вҖ” *not* inside `workspace/`. That keeps it clear of workspace re-initialization and makes it the first thing anyone sees when they open `~/.wolffish/`.

<Tip>
  This file is **managed by Wolffish** вҖ” the bundled copy is rewritten on every launch, so an app update always refreshes it to the current version. Treat it as read-only: local edits are replaced on the next launch. For custom agent behavior that should *persist*, use [`agents.md`](/configuration/agents-md) instead.
</Tip>

````
## 1. The idea

**Wolffish is a brain you own, not a chatbot you rent.**

It's a local-first, markdown-powered personal AI desktop agent. It runs natively
on macOS, Windows, and Linux вҖ” thinking, acting, and learning on the user's own
machine. The core experience works fully offline through Ollama; cloud models
(Claude, GPT, DeepSeek, вҖҰ) are optional quality upgrades, never requirements.

Three ideas hold the whole design together:

- **Markdown is truth.** Every fact the agent knows вҖ” its personality, its
  memory, its skills, its schedule вҖ” lives in plain markdown files you can open
  in any text editor. To change what the agent does, you *edit a file*, not
  redeploy code. The SQLite index (`cortex.db`) is a derived cache; delete it and
  it rebuilds from the markdown.
- **A brain, not a loop.** The runtime is modeled on human neuroanatomy: 15
  specialized modules (thalamus, hippocampus, amygdala, вҖҰ), each with one job,
  wired together by an event bus. Every message follows the *same* deterministic
  path through the *same* gates. The LLM is creative at exactly one point; the
  structure around it is predictable. Trust comes from that predictability.
- **You own all of it.** Everything Wolffish is lives in this one folder,
  `~/.wolffish/`. Back it up by copying a folder. Version it with git. Uninstall
  with `rm -rf ~/.wolffish/`. There is no hidden cloud state, no opaque embedding
  store, no vendor lock-in.

Built by Younes Alturkey. Tagline: *Wolffish. Bite through anything.* рҹҗә

## 2. The stack

| Layer | Technology |
|---|---|
| Desktop shell | Electron 39, `electron-vite` (three processes: **main** = Node, **preload** = bridge, **renderer** = React) |
| Frontend | React 19, TypeScript 5.9, Tailwind CSS 4 |
| Build | Vite 7, electron-builder |
| Search index | SQLite (`better-sqlite3`) with FTS5 full-text search |
| Event bus | `mitt` (typed pub/sub) |
| LLM providers | Pure `fetch()`, no vendor SDKs вҖ” Anthropic (Claude), OpenAI (GPT), DeepSeek & other OpenAI-compatible clouds, Ollama (local) |
| Messaging channels | `grammy` (Telegram), Baileys (WhatsApp) |
| File processing | Sharp (images), pdf-parse, mammoth (docx), xlsx |
| Speech | faster-whisper (speech-to-text), Kokoro neural TTS (text-to-speech) |
| i18n | i18next вҖ” English + Arabic (full RTL) |
| Logging | Pino |
| Scheduling | node-cron (driven by `heartbeat.md`) |

The **application code** lives in a separate repo (`wolffish-app`), under
`src/main/runtime/` (the brain), `src/renderer/` (the UI), etc. **This folder
(`~/.wolffish/`) is the runtime data** the installed app reads and writes. You
generally help the user here, in their data вҖ” not in the app source.

## 5. The brain вҖ” 15 modules

Each module maps a brain region to one runtime responsibility and (usually) one
place on disk. This is the mental model behind the folder layout.

| Module | Job | On-disk home |
|---|---|---|
| **Thalamus** | Resolve the selected Brain model and stream from it (no cascade) | вҖ” (talks to providers) |
| **RAS** | Score relevance, allocate the token budget | вҖ” (pure heuristic) |
| **Prefrontal** | Assemble the system prompt + plan the approach | `brain/prefrontal/` |
| **Cortex** | Fast full-text retrieval over all markdown | `brain/cortex.db` |
| **Hippocampus** | 3-tier memory: episodes вҶ’ consolidated вҶ’ knowledge | `brain/hippocampus/` |
| **Cerebellum** | Load & match capabilities (skills + plugins) | `brain/cerebellum/` |
| **Wernicke** | Parse LLM output into text + tool calls | вҖ” (pure parser) |
| **Broca** | Assemble & stream the final response to the UI | вҖ” (pure composer) |
| **Amygdala** | Safety-gate every tool call (safe/confirm/block) | вҖ” (uses SKILL patterns) |
| **Motor** | Execute tools, retry, log each step | `brain/motor/tasks/` |
| **Basal ganglia** | Record outcomes, learn preferences | `brain/basalganglia/` |
| **Hypothalamus** | Monitor vitals (context, tokens, RAM, disk) | вҖ” (samples runtime) |
| **Brainstem** | Run cron jobs + watch files for the index | `brain/brainstem/heartbeat.md` |
| **Corpus** | Typed event bus connecting every module | `brain/corpus/` (daily log) |
| **Insula** | Self-awareness / introspection on demand | reads motor + basalganglia + hippocampus |

The per-turn pipeline (same order every time):

```
thalamus вҶ’ ras вҶ’ prefrontal вҶ’ cortex вҶ’ hippocampus вҶ’ cerebellum
   вҶ’ [LLM CALL] вҶ’ wernicke вҶ’ amygdala вҶ’ motor вҶ’ broca
      вҶ’ hippocampus (save episode) вҶ’ basalganglia (record outcome)
```

вҖҰwith `corpus` emitting events throughout, `hypothalamus` monitoring, `brainstem`
running independently, and `insula` available on demand.

## 6. Available tools (capabilities)

A **capability** is a self-contained folder in `brain/cerebellum/` (dot-prefixed,
so hidden вҖ” `ls -a` to see them). Drop a folder in, and the agent learns a skill.

**Two kinds:**
- **Pure skill** вҖ” just a `SKILL.md` (instructions for the LLM using existing
  tools). Examples here: `.git`, `.planning`.
- **Plugin capability** вҖ” `SKILL.md` **+** `plugin/index.mjs` exporting executable
  tools. Most capabilities are plugins.

### The full catalog (24 capabilities)

| Category | Capability | What it gives the agent (representative tools) |
|---|---|---|
| **System** | `.shell` | Run shell commands вҖ” `shell_exec` (auto-elevation via native password dialog; `background:true` for servers) |
| | `.filesystem` | `file_read`, `file_write`, `file_patch` |
| | `.node` | Node.js runtime + npm вҖ” `node_check`, `node_install` |
| | `.package-manager` | Cross-platform packages (brew/winget/apt/dnf) вҖ” `pkg_install`, `pkg_check` |
| **Web** | `.web-search` | `web_search` (Brave), `web_fetch` (read a page) |
| | `.browser` | Headless Playwright automation вҖ” `browser_launch`, `browser_navigate`, `browser_click`, вҖҰ |
| | `.browser-extension` | Drive the user's **real, logged-in** Chrome/Brave via the extension вҖ” ~60 `ext_*` tools (`ext_navigate`, `ext_click`, `ext_set_value`, `ext_read_page`, `ext_query_selector`, `ext_get_url`, `ext_screenshot`, `ext_wait`, вҖҰ) |
| | `.cloudflared` | Expose a local service вҖ” `cloudflared_tunnel` |
| **Documents** | `.document` | docx/html/md вҖ” `document_read/create/modify/convert/merge` |
| | `.pdf` | `pdf_read/create/merge/split/modify/form/secure/compress` |

### Anatomy of a `SKILL.md`

```yaml
---
name: shell
description: Execute shell commands on the local system
triggers:            # words in a message that surface this skill to the planner

  - execute
  - terminal

  - name: shell_exec

**To add a capability:** create `brain/cerebellum/.<name>/SKILL.md` (and, for code,
`plugin/index.mjs`). On next launch the cerebellum loads it; its `triggers` make it
discoverable and its `tools` become callable. No app rebuild needed вҖ” that's the
markdown-as-truth payoff.

## 7. How things work

**The agent loop.** A message (from the desktop UI, Telegram, or WhatsApp) enters
the pipeline in В§5. The prefrontal reads `soul.md`, `user.md`, the agent
procedures, matched skills, and relevant memory, and assembles one system prompt.
The model streams a response; tool calls are parsed out, safety-checked, executed
with retries, and the result is streamed back. The turn is saved to an episode and
the outcome recorded.

**The Brain model.** You explicitly select one model вҖ” the **Brain**
(`config.llm.brain`, a `{ providerId, model }` pair) вҖ” in Settings вҶ’ Modes, and
that is the model the thalamus runs. There's no cascade and no automatic
substitution between providers: on a transient error the thalamus retries the
**same** Brain on a backoff schedule (using `net.isOnline()` for instant offline
detection) rather than falling through to a different one. To run offline, pick
Ollama as your Brain (or use the chat "Local Only" toggle / `llm.localOnly`).
Orchestrator mode (`llm.orchestratorMode: "orchestrator"`) optionally adds a
second explicitly-chosen model вҖ” the **Worker** (`llm.workerModel`) вҖ” that the
Brain delegates parallel work to; both are explicit choices, still no cascade.

**Three-tier memory.**
- *Episodes* (`hippocampus/episodes/`) вҖ” appended every turn, zero-latency, no LLM.
- *Consolidated* (`hippocampus/consolidated/`) вҖ” weekly LLM summaries, written by
  nightly compaction (configured in Settings вҶ’ Hippocampus вҶ’ Compaction, not in
  `heartbeat.md`).
- *Knowledge* (`hippocampus/knowledge/`) вҖ” durable facts, promoted from episodes or
  written by hand. The prefrontal pulls candidates from all three and the RAS keeps
  only what scores high enough for the current message.

**Safety (amygdala).** Every tool call is classified using the `danger_patterns` /
`confirm_patterns` in each `SKILL.md`: *safe* runs immediately, *confirm* needs user
approval, *destructive* is blocked. `config.json вҶ’ safety.bypassPermissions` can
auto-approve confirms; heartbeat (scheduled) jobs auto-approve by design.

**Background jobs (brainstem + `heartbeat.md`).** Each `##` heading in
`heartbeat.md` is a scheduled job; its body is the instruction, fed to the full
pipeline as if the user typed it. Schedule kinds: `Startup`, `Every (Nm)`,
`Every (Nh)`, `Hourly (mm)`, `Daily`/`Nightly (HH:MM)`, `Weekday (HH:MM)`,
`Weekly (Day HH:MM)`, `Monthly (DD HH:MM)`, and raw `Cron (expr)`. In a fresh
install every example is commented out вҖ” uncomment a block to activate it.

**Channels.** The desktop app, Telegram, and WhatsApp all share one brain. A
`TurnRunner` serializes turns (FIFO) so the channels don't collide; a `TurnRouter`
sends approval prompts back to whichever channel owns the active turn.

**Conversations & the context meter.** Each conversation JSON carries a
`contextMeter` (`contextTokens` vs `contextBudget`, e.g. 108 867 / 1 000 000) and a
`timeline` of structured events (`context.built`, `llm.response`, `safety.allowed`,
`segment.*`, `task.*`). Assistant messages are split into typed `segments`
(`text`, `tool_call`, `tool_result`, `active_model`, `turn_end`).

**Usage accounting.** Every model call appends a line to `usage/daily/<today>.md`
and to `usage/providers/<provider>.md` вҖ” timestamp, model, token counts
(`in` / `out` / `cw`=cache-write / `cr`=cache-read), and dollar cost. This is the
ground truth for "what am I spending?"

## 9. How to help the user navigate вҖ” recipes

Concrete playbooks for the things users actually ask.

**"Change how my agent talks / behaves."**
Edit `brain/identity/soul.md` (personality, tone, humor, hard rules). For one-off
procedural rules, add them to `brain/prefrontal/agents.md` (your overrides win over
the core procedures). Changes take effect on the next turn вҖ” no restart needed.

**"Remember this about me / my project."**
Put durable facts in the right `brain/hippocampus/knowledge/*.md`
(`people`, `projects`, `preferences`, `technical`, `decisions`) or in
`brain/identity/user.md`. Hand-edited knowledge is first-class вҖ” the agent reads it
exactly like promoted memories.

**"Why did the agent do that?"**
1. Open today's `brain/corpus/YYYY-MM-DD.log.md` вҶ’ see every tool call + output.
2. For a specific run, open the matching `brain/motor/tasks/TASK-*.md`.
3. To see *what the model was actually told*, read the timestamped dump in
   `brain/prefrontal/.debug/`. Cross-reference with the conversation JSON.

**"Add a new tool / skill."**
Create `brain/cerebellum/.<name>/SKILL.md` (+ `plugin/index.mjs` for code). Give it
`triggers` and `tools`, plus `danger_patterns`/`confirm_patterns` if it can do harm.
Reloads on next launch.

**"Run something on a schedule."**
Add a `##` job to `brain/brainstem/heartbeat.md` using one of the schedule kinds in
В§7. Keep the instruction in the body, plain prose.

**"What am I spending?"**
Sum `usage/daily/*.md`, or read a single provider's ledger in
`usage/providers/<provider>.md`. Each line already has the dollar cost.

**"It's acting weird / search seems stale."**
Delete `brain/cortex.db*` (the index rebuilds from markdown on launch). If a turn
half-wrote something, the markdown files are still the truth вҖ” fix them directly.

**"Switch model / add an API key / go fully offline."**
Edit `config.json вҶ’ llm`. Add a provider object (with `id`, `model`, `apiKey`) to
`llm.providers`, then point `llm.brain` at the `{ providerId, model }` you want to
run вҖ” that one model is the Brain. To go offline, select Ollama as your Brain (set
`llm.local.enabled = true` and point `brain` at it), or flip `llm.localOnly = true`
to force the local model regardless of the selected Brain. There's no `cloudPriority`
or `allowLocalFallback` cascade anymore. **Redact keys when you echo this file.**

**"Back up / move my agent to another machine."**
Copy `~/.wolffish/workspace/`. Skip `runtime/` (disposable Chromium state) and
`cortex.db*` (regenerates). The brain is just files.

**"Start over."**
`rm -rf ~/.wolffish/` returns a clean slate. There is nothing stored anywhere else.

## 10. How to write good prompts for Wolffish

Wolffish is an *agent with memory and tools*, not a stateless chatbot. Prompt it
accordingly.

**For everyday asks, just talk.** Natural language is enough вҖ” `triggers` route the
request to the right capability ("summarize this PDF", "commit my changes", "what's
on my calendar tomorrow"). Don't name tools; describe the outcome.

**Teach once, in markdown вҖ” don't repeat yourself every chat.** A standing
preference belongs in `soul.md` / `user.md` / `knowledge/*.md`, not re-pasted into
each prompt. The brain is editable for exactly this reason.

**For autonomous / long-running tasks, give it a spec.** The pattern that works
well here (visible across this user's real task history):

1. **ROLE** вҖ” the mode and hard environment constraints
   ("work only through the `ext_*` extension tools in my logged-in tab; never use a
   headless browser").
2. **MISSION** вҖ” the single, concrete goal.
3. **RULES** вҖ” the failure modes to avoid, stated as imperatives. Be explicit about
   what *counts as done* vs. what merely *looks* done.
4. **ENVIRONMENT** вҖ” exact tools/sites/selectors to use or avoid.
5. **PHASES / PER-ITEM LOOP** вҖ” break the work into ordered steps; finish one item
   fully before the next.
6. **EXIT CONDITION** вҖ” the precise, checkable state that ends the run.
7. **FINAL OUTPUT** вҖ” the shape of the deliverable (e.g. "a table with X, Y, Z").

**Bake in verification.** "Never assume success вҖ” confirm it." Tell it the *signal*
that proves a step worked (a URL containing `/comments/`, a file that now exists, an
HTTP 200), not just a tool returning `{"success":true}`. Many real failures are
"the tool said OK but nothing happened."

**Constrain the toolset when it matters.** If only one capability is valid, say so
and name what's forbidden вҖ” picking `.browser` (headless, logged-out) instead of
`.browser-extension` (the real session) is a recurring, expensive mistake.

**Make waits real.** If a task needs pacing (rate limits, polling), instruct an
explicit `ext_wait`/sleep вҖ” and say "a real pause, never simulated." The agent will
otherwise try to fast-forward.

**Right-size effort.** "Quick check" вҶ’ terse, one pass. "Be thorough / autonomous /
don't stop until done" вҶ’ it will plan, loop, and self-verify. The agent has loop-
awareness (it can detect dead loops), but a clear EXIT CONDITION is what keeps a long
run from spinning.

## 11. Safety & control

- **Amygdala gating.** Each tool call is *safe* (runs), *confirm* (asks first), or
  *destructive* (blocked), per the `danger_patterns`/`confirm_patterns` in each
  `SKILL.md`. There are no hidden guardrails beyond this вҖ” the rules are readable.
- **`config.json вҶ’ safety`.** `bypassPermissions: true` auto-approves *confirm*-level
  calls (convenience vs. caution вҖ” know which the user wants). `blockCredentials`
  toggles credential-pattern protection.
- **Scheduled jobs auto-approve.** Heartbeat tasks run unattended, so they don't
  pause for confirmation вҖ” be deliberate about what you put in `heartbeat.md`.
- **No artificial timeouts.** Tools run until they finish (except OOM protection);
  long-running commands are normal. Use `background:true` for servers/watchers.
- **Secrets live in `config.json`** (provider keys) and in OAuth/token state for
  `.google`, `.github`, `.notion`. Treat all of it as sensitive: never print,
  commit, or transmit it.

## 12. Golden rules for working inside `.wolffish`

1. **Markdown is truth; the DB is cache.** Fix behavior by editing markdown. Never
   hand-edit `cortex.db` вҖ” delete it to rebuild instead.
2. **Stay inside `~/.wolffish/`.** The whole agent is here by design. Don't scatter
   state elsewhere; uninstall must remain `rm -rf ~/.wolffish/`.
3. **Capability folders are dot-prefixed (hidden).** Use `ls -a` in
   `brain/cerebellum/`.
4. **`agents.md` вү  `agents.core.md`.** Put user customizations in `agents.md` (never
   overwritten, wins on conflict); leave `agents.core.md` alone.
5. **Don't touch `runtime/` or `.lock`.** Chromium plumbing and the single-instance
   guard вҖ” editing them only breaks things.
6. **Protect secrets.** Redact `apiKey`/token values whenever you display
   `config.json` or service state.
7. **Prefer reading before writing.** The corpus log, task transcripts, and
   `.debug/` prompt dumps usually answer "why" before you change anything.
8. **This guide is orientation, not the agent's manual.** For how the agent itself
   should behave, see `brain/prefrontal/agents.core.md`.
````
